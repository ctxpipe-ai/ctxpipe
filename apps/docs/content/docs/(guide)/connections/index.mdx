---
title: Connection overview
description: What ctx| connects today and how those connections feed the knowledge graph and MCP.
---

**Connections** are the inputs ctx| uses to build **organization-scoped context**
for humans and agents. Today the center of gravity is **Git**: your repositories
(and how you install the GitHub App) determine what code is cloned, indexed, and
ingested.

## Connector records in the backend

The API stores **integrations** (GitHub App installs, Confluence / Forge installs)
in a single **`connections`** table: each row has a `con_*` id, an org, a **type**
(`github` or `forge`), and a JSON **config** (credentials and type-specific
settings). An organization may have **more than one** connection of the same
type. The app’s **Connectors** page lists them via **`GET /:orgSlug/api/v1/connectors`**
(metadata only - not full config). Confluence **sync targets** point at a
**repository id** (a row under Repositories), not a free-text GitHub name.

## Primary connection: repositories

When you add a repository:

1. The **backend** stores it under your **organization** and triggers ingestion
   workflows.
2. The **codesearch** service clones the tree and maintains a **Zoekt** index for
   fast text and symbol search.
3. **LangGraph** ingestion graphs walk the codebase and write **structured claims**
   into the **graph database** (and related retrieval stores where configured).

All of that stays **tenant-isolated**: queries and tools enforce org boundaries.

## Secondary connections: agents and docs

- **Agents** connect via **MCP** at `/mcp?orgSlug=...` using API keys or OAuth.
  They do not pick an org inside the tool payload - the org is part of the URL.
- **Documentation and standards** in the repo (README, ADRs, `AGENTS.md`,
  skills directories) are part of what ingestion and `ctx_advisor` can reason
  over - there is no separate “upload PDF” product path documented in the OSS
  stack; treat the repo as the doc surface first.

## What is not a “connection” in the UI sense

Generic third-party wikis or drive folders are **not** first-class connection
types in the open-source docs today. If you need that, plan it as **custom
ingestion** or future product surfaces - see [Other connections FAQ](/docs/connections/faq).
