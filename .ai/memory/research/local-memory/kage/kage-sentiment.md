# Kage Sentiment

Sources searched: GitHub result, Reddit/HN/general search. No strong independent public discussion was surfaced.

Representative sources:

- GitHub: https://github.com/MushroomFleet/Kage

## Overall Sentiment

Kage has low visible public sentiment. The project fits a popular demand pattern - local-only graph memory - but there is not enough accessible independent feedback to judge sustained usability.

Confidence: low.

## Positive Themes

- Local-only privacy positioning is likely to resonate with coding-agent users.
- A dashboard helps users trust and debug memory state.
- Graph-based memory aligns with common dissatisfaction with raw vector stores.

## Negative Or Cautious Themes

- Little public evidence of adoption.
- Graph memory systems can look powerful but fail if agents write poor entities or relations.
- A Docker/dashboard stack is less lightweight than markdown or SQLite-only approaches.

## Perceived Pros

- Privacy-first local graph memory.
- More inspectable than hosted memory APIs.
- Potentially useful as an MCP reference implementation.

## Perceived Cons

- Unknown maturity.
- Unknown retrieval quality at scale.
- Not clearly coding-workflow-specific.

## Sentiment Summary

Kage is worth tracking as a local graph-memory design, but it should be treated as exploratory until there are more user reports or a deeper code audit.
