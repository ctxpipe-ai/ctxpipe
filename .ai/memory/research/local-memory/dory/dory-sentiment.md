# Dory Sentiment

Sources searched: Dory site, GitHub link, general web search. I did not find broad independent Reddit/HN/X feedback in accessible results during this pass.

Representative sources:

- Site: https://dory.deeflect.com/
- Repo: https://github.com/deeflect/dory

## Overall Sentiment

Independent sentiment is sparse, but the design aligns with sentiment patterns seen across other local-memory discussions: users trust human-readable, local, git/versionable memory more than opaque hosted stores.

Confidence: low.

## Positive Themes

- The product copy addresses a real multi-agent pain: explaining the same project/person/context to every tool.
- Markdown as source of truth is likely to appeal to users who are skeptical of vector databases and cloud memory.
- The five-verb surface is simple and memorable.
- Obsidian compatibility is a practical benefit for users already maintaining notes.

## Negative Or Cautious Themes

- No strong third-party usage reports surfaced.
- File-backed memories can still rot if agents write poor summaries.
- Concurrent write safety and conflict resolution need validation.
- Search quality depends on corpus hygiene and embeddings/index implementation.

## Perceived Pros

- Transparent and local.
- Easy to back up, diff, and edit manually.
- Shared across agents without vendor lock-in.

## Perceived Cons

- May require users to maintain a good information architecture.
- Less automatic capture than hook-based systems.
- Probably weaker for app-level user memory at scale.

## Sentiment Summary

Dory is a strong design reference for local-first, human-owned memory. The current public-sentiment weakness is not negative feedback; it is lack of visible adoption evidence.
