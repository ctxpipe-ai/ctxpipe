# Beads Content Model

Sources: https://github.com/gastownhall/beads, https://agentpatterns.ai/agent-design/beads-task-graph-agent-memory/, https://github.com/gastownhall/beads/blob/main/docs/FAQ.md, https://steve-yegge.medium.com/introducing-beads-a-coding-agent-memory-system-637d7d92514a

## What It Stores

Beads stores work items, not generic memories. The core memory object is an issue/task with graph relationships.

Stored content includes:

- title;
- description;
- status;
- priority;
- issue type, such as task, bug, epic, or message;
- assignee/claim state;
- dependency edges;
- parent/child hierarchy;
- audit trail;
- notes and close reasons;
- project-level memories via `bd remember`;
- messages and threaded message issues;
- discovered follow-up work.

## Semantics / Types It Looks For

Beads does not extract broad semantic memory from every conversation. It encourages agents to convert work state into explicit issue semantics:

- open work;
- blocked work;
- ready work;
- dependencies;
- blockers;
- epics and subtasks;
- duplicate/superseded/related work;
- discovered work;
- persistent project insights via `bd remember`;
- follow-up tasks.

Dependency edge types include `blocks`, `depends_on`, `relates_to`, parent-child relationships, `duplicates`, `supersedes`, and `replies_to`.

## Extraction Prompt

No LLM extraction prompt was found because Beads is primarily CLI/tool-mediated. The relevant "prompt" is the agent workflow instruction generated by `bd setup` / `bd onboard`.

Public README prompt snippet: https://github.com/gastownhall/beads#readme

The README's minimal agent instruction tells agents to use `bd prime`, `bd ready`, `bd show`, `bd update --claim`, `bd close`, and `bd remember`, and to avoid markdown TODO files for task tracking.

## How It Manages Memory Soup

Beads avoids soup by refusing to be a general memory bucket:

- every remembered work item has status and priority;
- ready work is computed from dependency state;
- hash IDs reduce branch/agent collisions;
- epics and child IDs group related tasks;
- `bd ready` gives the agent a bounded next-action list;
- `bd prime` injects workflow context and persistent project memories;
- compaction summarizes old closed tasks;
- Dolt/git-backed storage makes memory reviewable and mergeable;
- tasks can be closed, blocked, superseded, or related instead of duplicated in prose.

The main remaining soup risk is duplicate issue creation when agents ignore the workflow or create vague tasks. Beads addresses this more through agent discipline and graph queries than through semantic extraction.

## Notes For ctxpipe

Beads shows that coding-agent memory should store operational state directly. For ctxpipe, a task/work graph may be more valuable than a generic vector memory for current implementation continuity.
